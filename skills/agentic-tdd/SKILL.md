# Agentic Engineering TDD (RED - GREEN - REVIEW)

## 개요

Agentic Engineering TDD는 Coding Agent가 코드를 작성하기 전에 **사람이 먼저 계획을 검토**하고,
구현이 끝난 뒤에는 **사람이 결과를 리뷰**하는 human-in-the-loop 사이클이다.

기존 TDD의 Red-Green-Refactor와 다른 점: Agent가 자율적으로 다음 테스트를 계속 진행하지 않는다.
매 사이클마다 "계획 승인 → 구현 → 결과 승인"의 관문(gate)을 사람이 통과시켜야 다음 단계로 넘어간다.

핵심 원칙: **사람의 승인 없이는 다음 단계로 넘어가지 않는다.**

## 3단계

### 1. RED — 목표 설정 + Plan.md 작성

- 이번 사이클에서 달성할 목표(goal)를 정확히 정의한다 (모호하면 사람에게 먼저 질문한다).
- 목표, 구현 예정 테스트 케이스, 구현 접근 방식, 커밋 계획을 담은 `Plan.md`를 작성한다.
- **Plan.md 작성 후 코드를 작성하지 않고 멈춘다.** 사람에게 검토를 요청한다.
- 사람이 승인하면: `Plan.md`만 커밋한다 (RED 커밋). 코드는 아직 없다.
- 사람이 수정을 요청하면: Plan.md를 수정하고 다시 검토를 요청한다. 승인 전까지 반복한다.

Plan.md에 포함되어야 하는 것:
- 목표 (Goal)
- 구현 예정 테스트 케이스 목록
- 구현 접근 방식 개요
- 파일 위치
- 커밋 계획 (RED 커밋 / GREEN 커밋 시점)

### 2. GREEN — Plan대로 구현

- **승인된 Plan.md 범위 안에서만** 구현한다. Plan에 없는 기능·리팩토링·최적화를 임의로 추가하지 않는다.
- 테스트를 작성하고 구현한 뒤, 실제로 `pytest`를 실행해 통과하는지 직접 확인한다.
- 이 단계에서는 **아직 커밋하지 않는다.** GREEN 코드는 사람 리뷰 전까지 커밋 대기 상태로 둔다.
- 구현 중 Plan의 범위를 벗어나야 할 필요를 발견하면(예: 엣지 케이스 누락), 코드를 계속 진행하지 말고
  사람에게 알리고 Plan.md를 먼저 수정한 뒤 승인받는다 (RED로 되돌아감).

### 3. REVIEW — 사람 리뷰 후 커밋

- 구현된 코드와 테스트 결과를 사람에게 제시한다.
- 리뷰 시 확인할 것:
  - Plan.md에 명시된 것 **이상**으로 구현된 것이 있는가? (범위 이탈)
  - Plan.md에 명시된 것 중 빠진 것이 있는가?
  - 리팩토링이 필요한 부분이 있는가?
- 사람이 리팩토링을 요청하면 반영하고 다시 리뷰를 요청한다.
- 사람이 승인하면: GREEN 코드(구현 + 테스트)를 커밋한다 (GREEN 커밋).
- **사람 승인 없이 GREEN 코드를 커밋하지 않는다.**

## 커밋 규칙

| 시점 | 커밋 대상 | 조건 |
|---|---|---|
| RED | `Plan.md` | 사람이 Plan을 승인한 후 |
| GREEN | 구현 코드 + 테스트 | 사람이 REVIEW에서 승인한 후 |

커밋 메시지에는 단계를 명시한다. 예: `RED: Plan.md for bowling score cycle`, `GREEN: implement frame scoring per Plan.md`.

## Iron Law (철의 법칙)

1. **Plan.md가 사람에게 승인되기 전에는 구현 코드를 작성하지 않는다.**
2. **사람이 GREEN 결과를 리뷰하고 승인하기 전에는 커밋하지 않는다.**
3. Plan에 없는 범위는 그 자리에서 임의로 구현하지 말고, 먼저 Plan.md를 갱신해 재승인받는다.

이 셋 중 하나라도 건너뛰었다면 Agentic Engineering TDD가 아니라 그냥 TDD(또는 그 이하)를 한 것이다.

## 흔한 위반 패턴 (Red Flags)

- Plan.md 승인 답변을 받지 않고 바로 구현으로 넘어감
- Plan.md에 없는 헬퍼/최적화/추가 테스트 케이스를 "겸사겸사" 구현
- GREEN 구현 후 사람 리뷰를 요청하지 않고 바로 커밋
- 리뷰에서 지적받은 리팩토링을 반영하지 않고 그대로 커밋
- RED/GREEN 커밋을 하나로 합쳐서 커밋 (단계 구분이 사라짐)

## 검증 체크리스트

사이클을 완료로 표시하기 전에:

- [ ] Plan.md에 목표/테스트 케이스/구현 접근/커밋 계획이 모두 있다
- [ ] Plan.md에 대해 사람의 명시적 승인을 받았다
- [ ] RED 커밋(Plan.md만)이 별도로 존재한다
- [ ] 구현이 Plan.md 범위를 벗어나지 않았다
- [ ] `pytest`로 테스트 통과를 직접 확인했다
- [ ] GREEN 코드에 대해 사람의 리뷰(범위 이탈/리팩토링 필요성)를 받았다
- [ ] 리뷰에서 나온 수정 요청을 반영했다 (있었다면)
- [ ] 사람의 승인 후에만 GREEN 커밋을 했다

체크박스를 모두 채울 수 없다면, 그 단계로 돌아가 사람의 승인을 받아야 한다.
